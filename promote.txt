# Coder Agent 项目规划文档（类 Claude Code）

当前检查日期：2026-06-11

## 一、项目定位

本项目定位是一个 Claude Code 风格的 Coder Agent / Agent Runtime，不是通用聊天机器人，也不是纯 RAG 问答系统。

核心目标：

- 理解用户编程意图，自主规划任务。
- 优先通过工具搜索、阅读、编写项目代码（Tool-first，而不是 RAG-first）。
- 执行代码并根据结果修正，形成“写 -> 跑 -> 看结果”的闭环。
- 具备任务拆解与进度追踪（TodoList）。
- 自动压缩上下文，避免长对话撑爆上下文窗口（Context Compression）。
- 对高风险工具执行权限分级管理（Permission System）。
- 自动读取项目级 AGENT.md，理解项目规则（Project Context）。

设计重点仍然是：Agent Loop 本身并不复杂，真正困难的是 Context Management、Tool System、Permission System、Memory System。

## 二、七层设计与当前状态

### 1. Agent Loop（ReAct 循环）- 基础完成，需增强稳定性

当前 `core/runtime.py` 已实现 Function Calling 循环：

```text
Thought -> Action(tool call) -> Observation(tool result) -> ... -> Answer
```

已具备：

- `AgentRuntime.run()` 主流程。
- Native tools 多轮调用，`MAX_TOOL_ROUNDS=12`。
- 工具调用结果写回短期记忆。
- 支持 RAG、长期记忆、AGENT.md、TodoList 上下文注入。

待增强：

- 多个 tool_calls 同时返回时当前只处理第一个，需要补齐多工具调用支持。
- 工具调用错误恢复策略较弱。
- 工具结果只做字符截断，没有按 token 或文件结构压缩。

### 2. TodoList（任务拆解与追踪）- 基础完成，未自动化

当前已有 `core/todo.py`，并已接入 `main.py` 与 `context.py`。

已具备：

- `Task` / `TaskStatus` / `TodoList`。
- `add/start/complete/reset/block/cancel/insert/remove`。
- `format_for_prompt()` 注入给 LLM。
- CLI 命令：`/todo show | add | start | done | reset | clear`。
- `test_stage2.py` 已覆盖并通过。

当前问题：

- Agent 不会主动拆解任务，也不会自动更新任务状态。
- TodoList 现在更像“用户手动进度面板”，还不是 Claude Code 风格的自动规划能力。
- TodoList 未持久化，`/clear` 或重启后状态会丢失。

下一步建议：提供 `update_todo` 工具，或者在 runtime 收到复杂任务时先要求模型生成计划，再写入 TodoList。

### 3. Tool Use > RAG - 方向正确，但代码搜索工具不足

当前已有：

- `read_file`
- `write_file`
- `execute_code`
- `web_search`

这能完成基础读写执行闭环，但距离代码代理常用工具仍有差距。

待补齐：

- `grep`：正则搜索代码内容。
- `ls`：列出目录结构，并过滤 `venv/__pycache__` 等噪音目录。
- `edit_file`：基于行号或精确匹配的小范围编辑，比 `write_file` 全量覆盖安全。

说明：RAG 模块可以保留用于文档类知识检索；代码理解应优先走 `ls/grep/read_file`。

### 4. Context Compression（上下文压缩）- 未完成，仅有雏形

当前已有：

- `ShortMemory`：按消息数和 token 粗略截断。
- `LongMemory`：JSON 持久化 facts/summaries。
- `LongMemory.summarize_and_store()`：可保存简单摘要。
- `AgentRuntime.context_compressor` 扩展点。

当前问题：

- `summarize_and_store()` 默认只是统计“用户问了几个问题、助手回了几条”，不是 LLM 智能总结。
- `_check_and_compress()` 调用 compressor 后没有替换 `ShortMemory` 的旧消息。
- 没有保留“文件修改记录、错误日志、测试结论”等代码任务关键事实的结构化策略。

目标形态：

```text
if total_tokens > threshold:
    summary = llm_summarize(old_history)
    memory.messages = [summary_message] + recent_messages
```

### 5. SubAgent - 暂不做

第一版不建议引入 SubAgent。当前单 Agent 的工具、权限、上下文能力还没稳定，过早引入 SubAgent 会放大调度复杂度。

### 6. Permission System（权限分级）- 未真正接入

当前已有设计痕迹：

- `tools/base_tool.py` 定义了 `SAFE/CAUTION/DANGEROUS` 和 `risk_level`。
- `write_file` 标为 `CAUTION`。
- `execute_code` 标为 `CAUTION`，注释里提到 shell 模式应动态升级为 `DANGEROUS`。
- `AgentRuntime` 预留了 `permission_fn`。

当前问题：

- 没有 `core/permission.py`。
- `main.py` 没有创建或注入 PermissionManager。
- Runtime 执行工具时默认不会根据 `risk_level` 弹出确认。
- `execute_code(shell)` 仍可直接通过模型调用，只依赖少量黑名单拦截。
- `write_file` 只限制 `/dev /proc /sys /boot`，没有限制工作区边界。

建议分级：

- SAFE：`read_file`、`grep`、`ls`。
- CAUTION：`web_search`（可能泄露查询内容）、`write_file`、`execute_code(python)`。
- DANGEROUS：`execute_code(shell)`、删除文件、修改工作区外文件、安装依赖、网络写操作。

### 7. AGENT.md（项目级上下文）- 基础完成

当前已有：

- 根目录 `AGENT.md`。
- `core/project_context.py`。
- `main.py` 启动时读取 `AGENT.md`。
- `context.py` 每轮把项目规则注入 system context。
- `test_stage2.py` 已覆盖并通过。

待增强：

- 支持指定项目根目录，而不是只按当前 main.py 所在目录。
- 支持 AGENT.md 热更新或 `/reload` 命令。
- 避免 AGENT.md 过大时挤占工具结果和对话历史 token。

```text

│   ├── runtime.py             # Agent Loop + Function Calling
│   ├── message.py             # 统一消息结构
│   ├── memory.py              # ShortMemory + LongMemory + 简单自动总结
│   ├── context.py             # Prompt Context 构建
│   ├── todo.py                # TodoList 数据结构
│   └── project_context.py     # AGENT.md 读取与注入
├── llm/
│   └── deepseek_client.py     # DeepSeek API，OpenAI 兼容 Function Calling

│   ├── base_tool.py           # Tool 抽象基类 + risk_level
│   ├── registry.py            # Tool 注册中心
│   ├── file_tool.py           # read_file / write_file
│   ├── code_executor.py       # execute_code(python/shell)
│   └── web_search.py          # web_search(DuckDuckGo / Google CSE)
├── rag/
│   ├── embedding.py           # TextSplitter + BGE-M3
│   └── retriever.py           # 向量检索器
├── main.py                    # 终端对话入口
├── AGENT.md                   # 项目级上下文规则
├── test_tools.py              # 工具层测试

```

注意：当前项目目录中没有 `.git` 仓库，因此无法通过 git 状态判断改动历史。

## 四、核心数据流（目标版本）

```text
用户任务
  -> 注入 AGENT.md 项目规则
  -> Agent 自动创建 TodoList（当前尚未自动化）
  -> ReAct Loop
      -> ls/grep/read_file 搜索代码
      -> edit_file/write_file 修改代码（权限确认）
      -> execute_code 跑测试或验证（权限确认）
      -> 根据 Observation 继续修正
  -> Context 过大时压缩旧历史（当前尚未完成）
  -> Memory 更新 + TodoList 标记完成（当前 Todo 更新需手动）
  -> 最终答复
```

## 五、发现的主要问题与冲突

1. 规划文档状态落后于代码。

   旧文档把 `core/todo.py`、`core/project_context.py`、`AGENT.md` 标为待做，但这些已经存在并完成基础集成。

2. LLM 后端描述冲突。

   旧文档仍写 `llm/gemini_client.py`，实际代码主流程是 `llm/deepseek_client.py`；AGENT.md 也明确 DeepSeek 是主后端。

3. TodoList 能力描述过度。

   文档说“Agent 接到任务后先拆解并逐项标记”，但当前只支持 CLI 手动管理，LLM 没有工具或 runtime 逻辑去自动更新 TodoList。

4. Permission System 还不是可用功能。

   工具有 `risk_level` 字段，runtime 有 `permission_fn` 扩展点，但没有 PermissionManager，也没有在 main.py 中接入确认流程。

5. Context Compression 与 LongMemory 混在一起描述。

   当前 LongMemory 只是持久化摘要；真正的 compaction 应该替换短期历史中的旧消息，目前没有实现。

6. Tool-first 路线缺少关键工具。

   对代码项目来说，`grep/ls/edit_file` 比 RAG 更核心。当前只能读文件、写文件、执行命令，容易让模型用 `execute_code(shell)` 去代替安全工具。

7. 旧测试与文档曾有轻微漂移。

   `test_tools.py` 曾依赖 `promote.txt` 的具体文案。当前文档已保留兼容关键词，脚本测试已通过；后续仍建议测试文件读取结构，不要强绑定规划文案。

## 七、下一步实施顺序

建议按以下顺序推进，而不是先做 Web：

1. 补 `core/permission.py` 并在 `main.py` 注入 PermissionManager。
2. 增加 `grep`、`ls`、`edit_file` 三个工具，减少模型滥用 shell。
3. 给 TodoList 增加模型可调用的 `update_todo` 工具，完成自动任务追踪。
4. 实现真正的 Context Compression：LLM 摘要 + 替换旧短期历史。
5. 增强测试：Runtime 工具调用、权限拒绝、Todo 自动更新、压缩触发。
6. 第一/二阶段稳定后，再做 FastAPI + WebSocket。
