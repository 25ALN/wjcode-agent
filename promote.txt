# Coder Agent 项目规划文档（类 Claude Code）

当前检查日期：2026-07-13

## 一、项目定位

本项目定位是一个 Claude Code 风格的 Coder Agent / Agent Runtime，不是通用聊天机器人，也不是纯 RAG 问答系统。

核心目标：

- 理解用户编程意图，按任务复杂度选择执行策略。
- 简单问题使用 ReAct 工具循环快速处理。
- 复杂问题使用 Planning + ReAct：先规划，再执行，执行过程中根据工具观察继续重规划。
- 优先通过工具搜索、阅读、编写项目代码（Tool-first，而不是 RAG-first）。
- 执行代码并根据结果修正，形成“写 -> 跑 -> 看结果 -> 调整计划”的闭环。
- 具备任务拆解与进度追踪（TodoList）。
- 自动压缩上下文，避免长对话撑爆上下文窗口（Context Compression）。
- 对高风险工具执行权限分级管理（Permission System）。
- 自动读取项目级 AGENT.md，理解项目规则（Project Context）。

设计重点仍然是：Agent Loop 本身不复杂，真正困难的是 Context Management、Tool System、Permission System、Memory System，以及复杂任务下的 Planning 状态管理。

## 二、七层设计与当前状态

### 1. Agent Loop（ReAct 循环）- 已完成

当前 `core/runtime/agent_runtime.py` 已实现 Function Calling 循环，并通过 `core.runtime` 保持旧导入兼容：

```text
Thought -> Action(tool call) -> Observation(tool result) -> ... -> Answer
```

已具备：

- `AgentRuntime.run()` 主流程。
- Native tools 多轮调用，`MAX_TOOL_ROUNDS=12`。
- 支持单个 `function_call` 和多个 `function_calls`。
- 自动生成缺失的 tool call id，保证工具结果可回传给模型。
- 工具调用结果写回短期记忆。
- 支持 RAG、长期记忆、AGENT.md、TodoList、Planning、Scratchpad 上下文注入。
- 工具调用错误以 Observation 形式返回，模型可继续修正。
- 工具 schema 暴露使用 `core.intent` 的词边界匹配，避免 `latest` 误触发 `test`、`runtime` 误触发 `run` 这类英文子串误判。
- `run_events()` 是 Web/API 和兼容流式输出的 canonical 路径，`stream_run()` 只是薄包装，不再维护第二套执行逻辑。
- 普通问答追问会使用去工具协议化的历史上下文，避免上一轮 read_file/tool_calls 消息污染下一轮自然语言回答；“模块怎么样了/状态如何”这类状态问答默认不暴露工具；纯伪工具输出或纯检查前置语会针对通用 fallback 做一次严格普通回答重试/回退。
- 权限恢复会继续完成当前 assistant 返回的整批 `tool_calls`，确保每个 `tool_call_id` 都有对应 tool message 后才再次请求 DeepSeek，避免 HTTP 400 的 insufficient tool messages 错误。
- 项目/目录只读分析会进入收敛后的工具工作流：只暴露 `ls` / `grep` / `read_file`，最多执行 3 个项目分析工具观察；预算用完后运行时关闭工具 schema，并直接基于已有 Observation 生成确定性最终答案，避免开放式反复读文件导致模型超时。

当前限制：

- 工具结果仍主要依赖字符截断；`read_file` 已增加默认开头窗口和行号范围提示，但尚未实现按 token、文件结构或 diff 语义压缩。
- 复杂任务的规划步骤目前使用本地启发式计划，LLM 规划入口保留但默认关闭，避免额外模型调用干扰工具循环。

### 2. Planning + ReAct（复杂任务规划）- 已完成并接入

当前已有 `core/planning/manager.py`，并已接入 `core/runtime/agent_runtime.py`、`core/context/builder.py`、`main.py`。

职责：

- `PlanningManager.should_plan()` 判断任务是否复杂。
- 简单任务保持纯 ReAct。
- 复杂任务创建 `PlanState`，生成 3 到 6 个可执行步骤。
- 将计划同步到 TodoList，形成可见进度。
- 每次工具执行后读取 Observation。
- 发现错误、失败、权限拒绝、Traceback 等结果时追加修正和验证步骤。
- 写文件或编辑文件后自动补充验证步骤。
- 计划变化会在同一轮工具循环中重新注入模型上下文，避免只能下一轮才生效。
- `stream_run()` 通过 `run_events()` 启动 Planning，后续 Web/流式输出不会和普通 run 行为分裂。

当前形态：

```text
用户任务
  -> PlanningManager 判断复杂度
  -> 简单任务：ReAct Loop
  -> 复杂任务：生成 PlanState + 同步 TodoList
  -> ReAct Loop 执行工具
  -> Observation 反馈给 PlanningManager
  -> 必要时更新计划并继续执行
```

### 3. TodoList（任务拆解与追踪）- 已完成并自动化

当前已有 `core/todo/models.py`、`core/todo/store.py`、`tools/todo_tool.py`，并通过 `core.todo` / `core.todo_store` 保持旧导入兼容。

已具备：

- `Task` / `TaskStatus` / `TodoList`。
- `add/start/complete/reset/block/cancel/insert/remove/clear`。
- `format_for_prompt()` 注入给 LLM。
- CLI 命令：`/todo show | add | start | done | reset | block | cancel | remove | clear`。
- `update_todo` 工具，允许 Agent 自动维护任务列表。
- `.agent_todo.json` 持久化，重启后可恢复。
- PlanningManager 创建或更新计划时会同步 TodoList。
- `/clear` 和 `/todo clear` 会同时清理过期 Planning 状态。

说明：TodoList 既是用户可查看的进度面板，也是 Planning + ReAct 的执行约束之一。

### 4. Tool Use > RAG - 已完成关键代码工具

当前已有：

- `read_file`
- `write_file`
- `execute_code`
- `web_search`（DuckDuckGo HTML/Lite，免费且无需额外配置）
- `ls`
- `grep`
- `edit_file`
- `update_todo`

说明：RAG 模块可以保留用于文档类知识检索；代码理解应优先走 `ls/grep/read_file`，小范围修改优先走 `edit_file`，需要全量创建或覆盖文件时再使用 `write_file`。文件/项目工具的相对路径统一按 workspace root 解析，而不是进程当前目录。只读项目分析会限制为 `ls/grep/read_file`，并有 3 个工具观察硬预算；`read_file` 默认只显示文件开头窗口，模型需要后续内容时必须显式指定 `start_line/end_line`。

### 5. Context Management（上下文管理）- 已完成并持续增强

当前已有：

- `ShortMemory`：按消息数和 token 粗略截断。
- `LongMemory`：结构化 `MemoryItem` 持久化，兼容旧 `facts/summaries` JSON。
- `MemoryItem`：包含 `id/type/content/tags/source/importance/embedding/content_hash/superseded_by/metadata`。
- `LongMemory.search()`：基于当前查询做相关记忆检索，支持 type/tag/source 过滤。
- `LongMemory(embedder=...)`：可复用 RAG 的 `Embedding` / BGE-M3 dense vector；未启用 RAG 时自动回退到本地轻量词向量。
- `LongMemory.add_item()`：支持精确 hash 去重和本地向量相似去重。
- `MemorySummarizer`：本地 fallback 摘要会保留用户目标、工具调用、文件线索、错误、测试结论和助手结论。
- `core/context/scratchpad.py`：`Scratchpad` 显式记录目标、已确认事实、相关文件、已尝试操作、阻塞/风险和下一步。
- `core/context/compression.py`：`ContextCompressor` + `CompressionResult`，并保留 `core.compression` 兼容导出。
- `AgentRuntime.context_compressor`：达到阈值时触发压缩。
- Runtime 会用压缩摘要真实替换旧 `ShortMemory`，并把摘要写入 `LongMemory`。
- `AgentSession` 会把每轮用户可见对话持久化到 `.agent_sessions/<session_id>/history.json`，即使只有一轮会话也可在左侧历史列表恢复；空会话不会作为磁盘历史展示；history 为空但 runtime memory 有可见对话时会兜底回填；这与 LongMemory 摘要和 Context Compression 是分离机制。
- Runtime 注入长期记忆时会结合当前用户输入和 Planning 目标检索相关记忆，而不是固定注入最近几条。
- `main.py --rag` 会把 `Retriever.embedder` 传给 LongMemory，避免 RAG 和长期记忆各自重复加载 BGE 模型。

实现细节：

- 优先调用 LLM 对旧历史做摘要。
- LLM 不可用或失败时使用本地 fallback 摘要。
- 摘要会保留用户目标、工具调用、工具结果、错误信息、测试结论等关键信息。
- 如果压缩后 token 没有下降，会自动收紧摘要，避免“越压越大”。
- `main.py` 默认启用压缩器：`threshold_tokens=50000`，`keep_recent=12`。

当前 Context 组成：

```text
System Prompt
Project Context / AGENT.md
Todo Context
Planning Context
Scratchpad Context
Long Memory Context
RAG References
Tool Results
Chat History + User Message
```

Scratchpad 不是隐藏思考链，只保存可展示、可验证的工作笔记，用于减少复杂任务中重复读取和状态丢失。

压缩形态：

```text
if total_tokens > threshold:
    summary = llm_summarize(old_history)
    memory.messages = [summary_message] + recent_messages
```

### 6. SubAgent - 暂不做

第一版不建议引入 SubAgent。当前单 Agent 的工具、权限、上下文、规划能力已经能支撑阶段目标；过早引入 SubAgent 会放大调度复杂度。

### 7. Permission System（权限分级）- 已完成并接入

当前已有：

- `tools/base_tool.py` 定义了 `SAFE/CAUTION/DANGEROUS` 和 `risk_level`。
- `write_file` 标为 `CAUTION`。
- `edit_file` 标为 `CAUTION`。
- `execute_code` 标为 `CAUTION`，shell 模式会动态升级为 `DANGEROUS`。
- `web_search` 会动态升级为 `CAUTION`，因为联网搜索可能泄露查询内容。
- `core/permission/manager.py`：`PermissionManager` + `PermissionDecision`，并保留 `core.permission` 包入口导出。
- `AgentRuntime` 在执行工具前调用 `permission_fn`。
- `main.py` 在启用工具时创建并注入 `PermissionManager`。
- 文件工具相对路径统一按项目工作区解析；系统路径会被直接阻断。
- 工作区外读取、搜索、列目录和写入不会静默执行，默认进入权限确认；工作区外写入标记为 DANGEROUS，且可通过 `allow_dangerous=False` 策略直接禁止。

权限行为：

- SAFE：`read_file`、`ls`、`grep`、`update_todo`。
- CAUTION：`web_search`、`write_file`、`edit_file`、`execute_code(python)`。
- DANGEROUS：`execute_code(shell)`、删除文件、修改工作区外文件（默认需确认，可策略禁止）、安装依赖、网络写操作。

## 四、核心数据流（当前实现）

```text
用户任务
  -> 注入 AGENT.md 项目规则
  -> PlanningManager 判断任务复杂度
      -> 简单任务：纯 ReAct Loop
      -> 复杂任务：生成 PlanState，并同步 TodoList
  -> Context 注入 TodoList + Planning + Scratchpad 状态
  -> ReAct Loop
      -> ls/grep/read_file 搜索代码
      -> edit_file/write_file 修改代码（权限确认）
      -> execute_code 跑测试或验证（权限确认）
      -> 工具 Observation 写回 Memory
      -> Observation 交给 PlanningManager 判断是否重规划
      -> 计划变化时立即注入当前工具循环上下文
  -> Context 过大时压缩旧历史
  -> Memory 更新 + TodoList 持久化
  -> 最终答复
```

## 五、已解决的主要问题与剩余风险

1. 规划文档状态已与代码重新对齐。

   当前文档明确记录 `PlanningManager`、`TodoStore`、`PermissionManager`、`ContextCompressor`、`ls/grep/edit_file`、`update_todo` 的真实状态。

2. 纯 ReAct 已升级为 ReAct + Planning 混合模式。

   简单任务仍走轻量 ReAct；复杂任务会先规划、同步 TodoList，并在工具执行后根据 Observation 持续调整计划。

3. LLM 后端描述已统一。

   主流程是 `llm/deepseek_client.py`，Gemini 相关依赖仅保留兼容说明。

4. TodoList 自动化已补齐。

   当前既支持 CLI 手动管理，也支持模型通过 `update_todo` 工具自动维护任务列表，并通过 `.agent_todo.json` 持久化。

5. Permission System 已补齐。

   当前已有 PermissionManager、动态风险升级、工作区写入限制、系统路径阻断，以及 main.py 主流程接入。

6. Context Compression 已补齐。

   当前 ContextCompressor 会生成摘要并替换旧短期历史，同时把摘要写入 LongMemory。

7. Tool-first 关键工具已补齐。

   当前已有 `ls`、`grep`、`edit_file`，模型不再需要依赖 shell 完成常规代码搜索和小范围编辑。

8. 剩余风险。

   Planning 当前默认使用启发式步骤生成，稳定但不够智能；未来可以在不打断工具循环的前提下启用 LLM 规划或为计划步骤增加结构化状态。
